# Arkloop 投资人 Deep Research 方案包（中国市场 / 法律优先）

本文件用于驱动 AI 做“可引用、可验证、可转 PPT”的深度研究（Deep Research），并输出一套适合投融资路演的研究结论与验证计划。  
研究边界明确为：中国市场；切入点以律所/企业法务为第一场景，第二阶段再扩展为通用企业 Agent 平台能力。

0) 项目背景（已知事实，不要擅自扩写）
========================

Arkloop 是一个面向企业场景的 Agent Loop 系统，目标是把“聊天式交互 + 工具执行 + 审计/回放 + 多租户 + 流式事件”做成可商业化交付的平台。

当前（以仓库文档为准）可对外表达的底层不变量/差异化：
- 事件流是唯一真相：run_events 表达运行过程的一切（推送、审计、回放都基于同一事件表），并支持 SSE 流式消费与断线续传（after_seq 游标）。
- 控制面/执行面分离：API 只做编排与鉴权；Worker 执行 Agent Loop 与工具（规划方向明确）。
- Tools/Skills 强约束：Tool 有严格 schema、allowlist/policy、预算与风险等级；Skill 是可版本化的流程资产，强调可审计、可灰度、可回滚、可交付。
- 商业交付形态规划：SaaS + 线下机器（数据本地 + 云端模型）+ 线下 GPU（完全离线）；并支持 BYOK 与托管网关（Managed Gateway）两种模型接入方式。
- 安全与审计是一等公民：强调权限、可见性、导出、保留策略、以及管理员越权查看的强审计链路。

## C. 聚焦建议（法律优先 + 通用平台化）

以下是“建议结论”，用于你在投资人沟通中形成清晰主线；其中涉及市场与付费强度的部分必须用上面的 research 去验证与校准。

### 1) 一句话定位（建议用于 PPT 首页）

Arkloop：面向中国律所与企业法务的“可审计、可控工具执行、可交付的 Agent 工作台”，以事件流为底座，把 AI 从“对话”升级为“可追责的生产力系统”。

对标思路（表达方式，不是结论）：  
不是“又一个聊天机器人”，而是更接近“企业级工作流系统 + AI 执行层 + 审计与回放”，先在法律这种高风险场景立住，再把可交付能力外溢到其他行业。

### 2) 为什么法律是第一切入点（要点）

- 法律交付天然要求“证据链/引用/可解释/可追责”，与 Arkloop 的 `run_events`、审计与回放机制天然匹配。
- 客户通常对“数据安全、越权、外部发送、留痕”更敏感，愿意为可控性付费（需通过访谈与试点验证付费强度）。
- 法律场景的交付物形态稳定（合同审查意见、研究 memo、尽调清单等），更适合沉淀为可版本化的 Skill 资产，形成复用与交付壁垒。

### 3) 建议优先押注的 3 个法律场景（以“高频 + 高价 + 可量化”为准）

1. 合同审查与修订建议（企业法务 + 律所都可）
   - 成交点：条款级引用、风险分级、修改建议可追踪；输出可直接用于内部审计与对外交付。
   - 可量化指标：审查耗时降低、问题漏检率下降、交付一致性提升、复用模板数量增长。

2. 法律研究与检索（研究 memo / 法条与案例引用）
   - 成交点：要求“引用可追溯 + 来源可信 + 可复算”；对“只会聊天、不留证据”的产品容忍度低。
   - 可量化指标：研究时间降低、引用错误率下降、交付物可复用率提升。

3. 尽调与材料整理（多文档抽取、清单核对、问题追踪）
   - 成交点：数据范围广、协作链条长、交付压力大，适合用 Agent 做批处理与一致性核对；输出需要审计与回放。
   - 可量化指标：尽调周期缩短、问题清单覆盖率、返工次数下降。

### 4) 研发优先级（对齐现有工程路线图的“可卖点”补齐）

为了让法律 wedge 真的能卖、能交付、能续费，建议把研发优先级按“成交阻塞点”排序（不以炫技为目标）：

1. 可审计与可交付（必须优先）
   - 交付物必须能导出、可追溯（输出与引用材料的关联、变更记录、审批与 review 记录）。
   - 事件流的结构化：对工具调用、拒绝、预算超限等形成稳定事件类型与可检索字段（便于企业审计与问题定位）。

2. 权限、安全与部署形态（决定“能不能进入客户环境”）
   - 明确并实现：谁能看明文、谁能导出、谁能做高风险工具调用；越权必须强审计。
   - BYOK/托管网关/私有化的策略与产品化包装（对中国市场尤其关键，尽早在 research 里验证客户偏好）。

3. Skills 资产化（决定“能不能规模化交付”）
   - 把法律场景的流程沉淀为可版本化 Skill（输入/输出 schema、工具 allowlist、预算、review policy）。
   - 让 Skill 成为可售卖单元：行业包、团队包、私有部署包（具体定价方式由 research 验证）。

### 5) 商业化路径（建议叙事）

- 短期：法律 wedge 拿下 3-5 个设计伙伴，通过试点形成可复用的 Skill 与交付模板，验证付费与续费驱动因素。
- 中期：以“可审计的 Agent 执行层 + 技能包”为核心，形成 SaaS 起量与私有化高客单并行的产品线（部署形态取决于 research 结论）。
- 长期：把法律场景沉淀的能力（审计、权限、review、工具策略、skills 资产化）产品化为通用企业 Agent 平台能力，扩展到财务审计、内控、采购、合规等高风险领域。

### 6) 12-18 个月里程碑（用验证指标写，便于投融资叙事）

以下是“建议目标”，不是现状陈述；具体数值需结合研究与试点迭代：

- 0-3 个月：完成不少于 30 次高质量访谈；拿到 3 家设计伙伴试点意向；明确 1 个最强付费 ICP 与部署偏好分群。
- 3-6 个月：跑通 3 个试点并形成可复用交付包；至少 1-2 家转付费；沉淀 5-10 个可版本化法律 Skill。
- 6-12 个月：形成稳定的定价与交付方法论；建立渠道试点（至少 1 条）；复购/续费指标开始可追踪。
- 12-18 个月：在法律场景形成可复制的销售与交付闭环，并验证平台化外溢的第二曲线（优先选择同样强审计诉求的行业）。

## 使用建议（最省时间的推进方式）

1. 先用 Prompt 做桌面研究，产出“PPT 目录 + 关键不确定性清单”。
2. 立刻并行启动访谈与试点（不要等研究做完），用真实采购链路反证桌面结论。
3. 每 1-2 周把研究结论回写到“差异化叙事 + MVP 卖点清单”，避免研究变成信息堆积。
