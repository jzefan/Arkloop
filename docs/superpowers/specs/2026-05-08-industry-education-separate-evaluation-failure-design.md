# 分别评估全失败报错修复设计

**目标**：修复“双高产教融合评估”在“分别评估”模式下过于笼统的全失败报错，定位并修正至少一个模型成功时仍被判定为全失败的根因，同时向用户展示可操作的失败摘要。

## 背景

当前 `src/personas/industry-education-index/agent.lua` 在等待评估子智能体结果后，如果 `evaluations` 为空，会统一输出：

- `所有评估模型均未成功返回有效结果，请重试或更换模型。`

这个报错无法区分：

1. 子智能体调用失败或超时
2. 子智能体有输出，但输出为空
3. 子智能体有输出，但不符合评估 JSON 契约
4. “分别评估”模式下存在成功结果，但成功结果在主智能体侧被错误丢弃

根据 `src/personas/industry-education-index/prompt.md`：

- `综合评估` 和 `分别评估` 中，只要至少一个评估员成功，就应当降级继续。
- 只有所有评估员都失败时，才应停止并建议用户重试或更换模型。

因此当前实现至少存在可观测性不足，且可能存在“分别评估”成功结果被误判为失败的根因。

## 现状观察

重点实现位于 `src/personas/industry-education-index/agent.lua`：

- `validate_evaluation`：负责校验评估员输出 JSON 是否满足契约
- `wait_for_evaluators`：负责等待每个子智能体并筛选有效评估结果
- 主流程：当 `#evaluations == 0` 时输出统一失败文案

当前问题在于：

- `validate_evaluation` 只返回成功值或 `nil`，未暴露失败原因
- `wait_for_evaluators` 虽然记录了 `failures`，但最终用户报错未消费这些细节
- “分别评估”模式下无法从最终输出判断到底是全部失败，还是成功结果在校验过程中被过滤

## 设计原则

1. 保持最小改动：只修改 Lua 编排与相应测试，不改前端、API、persona 产品面。
2. 先保留证据，再做判断：每个模型失败都要留下可归因摘要。
3. 与产品语义对齐：`分别评估` 下只要有一个合法评估结果，就继续生成报告。
4. 不扩大范围：本次不引入新重试机制，不重写 evaluator prompt，不重设计报告结构。

## 方案

### 方案选择

采用“修根因 + 保留降级信息”的方案：

- 排查并修正 `分别评估` 模式下有效结果被误判的路径
- 将失败从“黑盒全失败”改为“带模型级原因摘要的全失败”
- 保持现有产品流和主流程结构不变

### 具体实现

#### 1. 扩展 `validate_evaluation`

将 `validate_evaluation` 从“返回评估对象或 `nil`”扩展为“返回评估对象，或返回 `nil + reason`”。

原因字符串至少覆盖：

- 输出为空
- JSON 解析失败
- 缺少 `dimensions`
- 维度数量不对
- 维度顺序错误
- 分数不在允许范围

这样主流程可以区分“模型没返回”和“模型返回了坏格式”。

#### 2. 在 `wait_for_evaluators` 中保留模型级失败摘要

对每个子智能体，统一记录：

- 模型标签
- 失败类别（子智能体失败 / 空输出 / 校验失败）
- 简短原因

若子智能体成功且通过 `validate_evaluation`，则照常加入 `evaluations`。

若失败，则加入 `failures`，供最终错误输出使用。

#### 3. 调整 `#evaluations == 0` 时的最终错误文案

当没有任何有效评估结果时，输出：

- 总结句：没有任何模型返回有效评估结果
- 附加摘要：逐个模型列出失败原因
- 收尾建议：请重试或更换模型

文案应让用户区分：

- 模型超时/执行失败
- 模型未返回内容
- 模型返回格式不符合要求

#### 4. 保持“部分成功即可继续”语义

在 `分别评估` 下，只要 `evaluations` 非空，就继续现有计算与报告生成逻辑。

如果最终发现根因是“成功结果被错误过滤”，修复应限制在判定逻辑本身，不改变后续 pipeline。

## 非目标

本次不包含：

- 修改前端表单或新增前端错误态
- 改动 API 返回结构
- 重写 `industry-education-evaluator` 或 `industry-education-synthesizer` prompt
- 设计新的“分别评估”报告版式
- 加入自动重试、回退模型、更多容错编排

## 测试设计

在 `src/services/worker/internal/executor/lua_test.go` 补充覆盖：

1. `分别评估` 下一个模型成功、一个模型失败：应继续，不出现全失败报错。
2. `分别评估` 下所有模型都返回无效 JSON：应报全失败，并包含至少两个模型的失败摘要。
3. `分别评估` 下所有模型都超时、失败或无输出：应报全失败，并包含失败摘要。
4. 回归确认：`综合评估` 与 `单模型评估` 现有行为不变。

## 验收标准

1. `分别评估` 下只要任一模型返回合法评估 JSON，就不会触发“所有评估模型均未成功返回有效结果”这一全失败路径。
2. 当全部失败时，最终输出必须包含模型级失败摘要，而非单一句泛化报错。
3. `综合评估` 和 `单模型评估` 行为不回归，仍满足 `prompt.md` 中的既有约束。
4. 改动范围限制在 Lua persona 编排与对应测试文件内。

## 风险与缓解

- 风险：失败摘要过长，影响用户可读性。
  - 缓解：每个模型只保留一条短原因，保持紧凑。
- 风险：现有测试桩不足以表达“成功但 JSON 无效”的场景。
  - 缓解：在 `lua_test.go` 中使用最小伪输出构造明确失败类型。
- 风险：根因在 evaluator prompt 与主校验契约不一致。
  - 缓解：先用测试精确暴露不一致；只有证据明确时才最小修正契约或提示，不主动扩大范围。
